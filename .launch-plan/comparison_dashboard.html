One-paragraph TL;DR

The two-model loop found real launch bugs that no single pass surfaced. GPT-5.5 v4 caught a hard-coded fake-stats array in Hero.tsx:5 ("12,000+ charts", "★ 4.8 from 340+") that violates the no-fake-proof rule. GPT-5.5 v6 audit of v5 caught five more — off-by-one in burst test, Ziina intent-reuse semantics, bypass E2E false-green, fake proof on the pricing page, and the fact that Upstash absence breaks locks not just rate limits (serverless = duplicate paid reports on double-click). Real launch bugs found by the loop

These are concrete, file-level issues that survived earlier passes and would have shipped to production without the validation loop. Each was caught by adversarial passes (v4 or v6) after Opus did not flag them.

Critical — caught in v4

Hero hard-coded fake stats

File: src/components/landing/Hero.tsx:5

TRUST_STATS = [{"value": "12,000+", "label": "charts generated"}, {"value": "★ 4.8", "label": "from 340+ seekers"}, ...]

Neither "12,000+ charts" nor "★ 4.8 from 340+" is tied to live instrumentation. Under the "no fake proof" rule (consensus from v2 onward), this violates a bright-line bans the platform agreed on by v2. Three plans missed it. v4 caught it.

Fix: Remove or replace with provable claims (Swiss Ephemeris, Lahiri ayanamsa). Sub-15 minutes of work.

Critical — caught in v4

Citation parser structurally requires verse numbers

Files: src/lib/reports/postProcess/extractCitations.ts:16 + src/components/report/ScriptureFootnotes.tsx:47

The parser regex requires all three of SOURCE:CHAPTER:VERSE. The UI prints "Ch." and "v." unconditionally. v3 said "audit + change prompt if metadata bad" — but even if the prompt is changed to allow source-only, the parser drops anything that doesn't have all three fields. v4 escalated to code change required.

Fix: Patch regex to accept SOURCE, SOURCE:CHAPTER, SOURCE:CHAPTER:VERSE. Patch UI to render "Ch." only when chapter present, "v." only when verse present. Remove "scholarly editions" footer text. Estimated 30–45 min.

Critical — caught in v6

Fake proof beyond Hero

File: src/app/pricing/page.tsx:166

The pricing page hard-codes "Chosen by 60% of our seekers" — exactly the same class of unsupported claim as the Hero counter. v4 caught the Hero but didn't site-sweep. v5 inherited the narrow Hero-only fix. v6 audit found the pricing claim. Also: "Most Popular" labels at pricing/page.tsx:63 and landing/Pricing.tsx:143 imply popularity data.

Fix: Site-wide sweep. Replace popularity claims with product proof. ~15 minutes additional.

High — caught in v6

Bypass E2E spec false-green

File: e2e/report-generation-bypass.spec.ts:72,77,78

The spec accepts status="error" with non-empty generation_error as a pass. v5 listed npm run test:e2e:bypass as a launch gate. Green CI doesn't mean successful report. The spec is useful diagnostic behaviour but is the wrong gate for "paid report works."

Fix: Add a separate hard assertion: at least one successful paid/bypass report with non-empty days array, no error status. Estimated 20 min.

High — caught in v6

Upstash absence breaks locks (not just rate limits)

Files: src/lib/redis/locks.ts:3,19,22 + src/lib/redis/client.ts:8

When Upstash env vars are absent, getRedis() returns null and locks fall back to localLocks (in-process Map). In serverless production, different invocations run in different processes — so "local locks" are effectively no locks at all. A user double-clicking "Generate" could spawn two paid generations charged to Ziina once. v5 rated this Medium. v6 correctly elevated to High.

Fix: Pre-launch hard test that locks are Redis-backed (acquire twice, verify second fails). If env missing, no-go.

Medium — caught in v6

Refund page support-email split-brain

File: src/app/refund/page.tsx:63,64,81

Refund callout tells users to email hello@vedichour.com; the refund CTA on the same page uses support@vedichour.com. Either both inboxes exist and are monitored or one is broken. Either way: confusing on launch day when actual refund requests come in.

Fix: Pick one. 5 min edit.

Medium — caught in v6

Burst test off-by-one acceptance

Files: src/lib/api/rateLimit.ts:48 + src/app/api/reports/start/route.ts:35,236,241

v5 §3.9 says 3 same-user starts should return {202, 202, 429}. But the limiter allows count <= limit, so 3 should all return 202; the 4th returns 429. QA running v5's test as written would falsely declare the limiter broken and waste launch-window recovery time.

Fix: Update acceptance: 3 same-user gets {202, 202, 202}; 4th gets 429. 1 min edit.

Medium — caught in v6

Ziina create-intent reuses pending intent

File: src/app/api/ziina/create-intent/route.ts:94,99,112,120

v5 §3.9 says "5 sequential Ziina create-intents in 30s should produce 5 distinct intent IDs". But the route intentionally reuses any pending intent for the same user/report/plan within 90s. Test as written would falsely declare Ziina broken.

Fix: Either use 5 distinct report IDs or change acceptance to "first call creates, next 4 calls reuse the same intent." 5 min edit.

Medium — caught in v6

Production bypass smoke assumes BYPASS_SECRET

Files: src/lib/api/requireAuth.ts:8,16,40,42 + scripts/verify-report-generation-env.mjs:82,83

v5 schedules a production bypass report after deploy. But bypass auth is disabled when BYPASS_SECRET is unset, and the env verifier only warns about missing bypass (does not fail). If prod doesn't have BYPASS_SECRET, the smoke silently can't run.

Fix: Either set BYPASS_SECRET in prod (with audit logging) or skip the prod bypass smoke and rely on a real low-value paid Ziina flow.

Pattern observations

  • GPT-5.5 is observably better at file-level / line-level repo reading. Every claim came with a path:line. Three real launch bugs were caught by GPT-5.5 (v4: Hero, citation, footer; v6: pricing, locks, refund). Zero were caught by Opus alone.
  • GPT-5.5 is observably better at ruthless triage. v2's 8-item P0 was cleaner than v1's 14-item P0; v4 pruned v3 further; v6 found stale items in v5.
  • GPT-5.5 is observably better at catching false-greens in test specs. v6 caught the bypass E2E pattern. Opus listed the same command as a launch gate without noticing the spec accepts errors.
  • Opus is observably better at operational depth. Cost ceilings, support runbooks, performance budgets, jurisdictional compliance all came from Opus v3 — items GPT-5.5 didn't propose despite having repo access.
  • Opus is observably better at strategic narrative. The five-paragraph "what each model is observably better at" framing came from Opus. The bright-line "no fake proof" rule that Opus added wasn't in v1 — but the rule itself was GPT-5.5's, and Opus had to be told.
  • Self-rating bias is large and one-sided. Opus v5 self-scored 89; GPT-5.5 v6 audit scored 76. Opus v3 self-scored 88; GPT-5.5 v4 audit scored 79. Both delta-9 to delta-13. GPT-5.5 self-rates conservatively (v2=86, v4=85).
  • Both passes are necessary. Neither model alone produced the final v5 plan. v5's biggest weaknesses (false-green gates, fake proof beyond Hero, Upstash severity) were caught only by adversarial GPT-5.5 in v6.

Single most important sentence

The launch goal is not "finish the roadmap" or "build the world's top 0.001% product in 18 hours." It is: no user pays Ziina and receives an ungrounded, broken, unobservable, or unrecoverable Vedic astrology report. Everything downstream serves that one goal.

Final verdict

Should you launch Monday 06:00 IST?

Yes — but only if v5 is patched with v6's 7 corrections first. Estimated patch effort: 90–120 minutes of focused work. If those patches aren't done by Sun 22:00 IST, fall back to Mon 18:00 IST.

The validation loop did its job. It found three real launch bugs that would have shipped to production:

  1. Hero hard-coded fake stats — would have damaged trust on the first viewing.
  2. Citation parser pattern lock — would have either dropped valid source-only citations or rendered "Ch.undefined" / "v.undefined" placeholders.
  3. Pricing page "Chosen by 60%" claim — same fake-proof violation as Hero, only caught after a third pass.

It also found four operational landmines:

  • Bypass E2E test passes on report failure ("false-green").
  • Upstash absence breaks shared locks → duplicate paid generations on double-click.
  • Refund page support email inconsistency (hello@ vs support@).
  • Off-by-one in burst test acceptance criteria.

What the loop reveals about the two models

  • Both models are strong, in different ways. Opus 4.7 is the better strategist and narrative author. GPT-5.5 xhigh is the better repo-reader and triage agent.
  • Neither model alone is sufficient. Opus's v1 would have shipped 14 P0 items in 48h, most of which were unnecessary or wrong. GPT-5.5's v2 was tight but missed operational nets (cost, support, compliance) that Opus added in v3.
  • The most expensive bias is self-rating. Opus consistently self-scored ~10 points above the adversarial pass. GPT-5.5 self-scored conservatively.
  • Adversarial passes find bugs that collaborative passes miss. Every concrete file-level bug came from a GPT-5.5 audit pass, not from Opus's authoring or revision passes.
  • Six iterations was not over-engineered for this scale of decision. Stopping after v4 would have missed the pricing-page fake proof, the bypass E2E false-green, and the lock-fallback severity. v5 alone would have shipped with a false sense of confidence.
